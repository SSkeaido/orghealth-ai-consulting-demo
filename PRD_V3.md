# OrgHealth V3 联合诊断工作台 PRD

> 文档版本：1.0  
> 产品版本：V3  
> 更新日期：2026-07-22  
> 当前阶段：前端交互原型

## 1. 产品定义

V3 用于处理企业同时发生多个经营难点的真实场景。用户可在专项视角或 CEO 视角下选择最多 3 个经营事件，指定一个主事件；系统合并并去重核心题，优先展示能够同时解释多个事件的交叉题，再通过关联确认和补充信息形成联合咨询路线。

一句话定位：不是分别诊断三个问题，而是判断多个经营事件是否来自同一个系统机制，并据此设计联合咨询路径。

## 2. 为什么需要 V3

V2.1/V2.2 的单事件流程适合清晰入口，但企业问题通常同时出现。例如“客户流失、交付延期、关键人才离开”可能不是三个独立问题，也不能简单叠加三个咨询产品。

V3 需要解决：

- 多个事件的题项重复，用户不应重复回答；
- 某些题项能同时支持多个事件，应优先验证；
- 必须区分主事件、并发事件和候选关联事件；
- 事件间可能是因果、相关、依赖或共同结果；
- 路由应形成有顺序的联合方案，而不是咨询模块购物车。

## 3. 目标与非目标

### 3.1 产品目标

- 支持一次选择 1—3 个经营事件；
- 支持专项视角和 CEO 视角，但一次诊断只能使用一个视角的事件库；
- 合并、去重并排序核心题；
- 优先验证多事件共享的交叉题；
- 第一轮输出多个初步方向但不显示置信度；
- 通过补充关联信息形成联合置信度和咨询路线；
- 明确展示事件关系仍是待验证假设。

### 3.2 非目标

- 不允许无限选择事件；
- 不跨 V2.1 和 V2.2 事件库混选；
- 不把并发关系直接解释为因果关系；
- 不把多条咨询路线简单相加后打包销售；
- 当前原型不包含真实因果推断、模型训练和顾问协作后端。

## 4. 核心概念

| 概念 | 定义 |
|---|---|
| 诊断视角 | 专项视角或 CEO 视角，决定可选事件库 |
| 主事件 | 用户当前最想优先解决、用于组织诊断叙事的事件 |
| 并发事件 | 与主事件同时发生、但关系尚未验证的已选事件 |
| 核心题 | 已选事件直接映射的第一轮题项 |
| 交叉题 | 同时被两个或更多已选事件引用的核心题 |
| 候选关联 | 系统根据事件映射提出、等待用户确认的相关现象 |
| 补充题 | 用户确认关联后动态调用的题项 |
| 联合置信度 | 联合咨询方向可靠程度，不是健康度或严重程度 |

## 5. 核心流程

```text
填写企业画像并选择诊断视角
  → 选择 1—3 个经营事件
  → 指定主事件
  → 合并、去重核心题
  → 交叉题优先回答
  → 输出多个初步咨询方向（无置信度）
  → 确认候选关联事件
  → 动态补充关联题项
  → 更新事件关系假设
  → 生成联合置信度与咨询路线
```

## 6. 功能需求

### 6.1 联合诊断入口

- 输入企业名称、行业和规模；
- 选择专项视角或 CEO 视角；
- 切换视角时清空已选事件和主事件，避免跨库混合；
- 最多选择 3 个事件；
- 默认将第一个选择的事件设为主事件；
- 用户可在底部事件组合栏重新指定主事件；
- 企业名称为空或没有主事件时不能开始。

### 6.2 合并核心题

- 汇总所有已选事件的核心题；
- 按题号去重；
- 按支持事件数量降序排列；
- 交叉题展示支持事件数量和事件名称；
- 每个题项只回答一次；
- 全部完成后才能进入联合工作台；
- 此阶段不计算或显示置信度。

### 6.3 联合概览

展示主事件、并发事件、多个初步咨询方向和候选关联事件。

方向区域要表达“共同解释优先”：系统先寻找是否存在能够同时解释多个事件的机制，再考虑拆分为独立专项。

### 6.4 候选关联去重

当多个已选事件提出同名关联事件时：

- 合并为一个候选项；
- 合并其补充题项并去重；
- 保留提出该候选项的来源事件；
- 用户确认一次即可。

### 6.5 补充证据

- 只展示已确认关联对应的题项；
- 排除已经在核心题阶段回答的题；
- 允许暂不填写；
- 至少回答一项新增补充题后，才显示联合置信度；
- 联合置信度旁必须说明“不表示问题严重性”。

### 6.6 事件关系

当前原型展示三层：主事件、并发事件、已确认关联事件。

正式版关系边应支持：

- 可能导致；
- 可能由其导致；
- 共同结果；
- 相互强化；
- 前置依赖；
- 仅相关，尚无因果证据。

任何关系默认是待验证假设，除非有明确证据和顾问确认。

### 6.7 联合咨询路由

路线生成必须考虑主事件优先级和多事件共同机制。输出最多 5 条候选模块，并明确：

- 联合优先模块；
- 相邻候选模块；
- 推荐置信度；
- 支持该路线的事件和题项；
- 模块之间的前后依赖；
- 哪些事件可能通过同一项目共同改善；
- 哪些事件需要拆分独立项目；
- 需要人工顾问确认的部分。

## 7. 推荐与排序原则

V3 推荐不能采用“每个事件各推荐一个项目，再按分数相加”。正式算法分为四步：

1. 候选生成：由事件映射、题项回答和咨询服务适用条件产生候选；
2. 共同机制识别：寻找能解释多个事件的共享题项和依赖关系；
3. 路线排序：综合严重程度、业务影响、战略重要性、紧迫性、改善可行性、证据可信度和服务适配度；
4. 组合编排：确定先做、并行做、后做和暂不做的咨询模块。

当前原型中的联合置信度与路线百分比是交互演示公式，尚未经过真实项目数据验证，不能作为正式咨询判断。

## 8. 当前原型范围

### 已实现

- 专项/CEO 诊断视角切换；
- 1—3 个事件选择和主事件指定；
- 核心题合并去重；
- 交叉题优先排序和来源标识；
- 多个初步方向；
- 候选关联事件合并、确认与补充题调用；
- 延迟显示联合置信度；
- 联合概览、补充证据、事件关系和咨询路由页面；
- 响应式布局和基础动态效果。

### 未实现

- 真实因果关系判定和关系编辑；
- 路线模块依赖图与项目组合编排；
- 顾问复核、批注和版本记录；
- 企业历史事件和跨周期比较；
- 后端存储、账号、权限、支付与子 Agent 调度；
- 推荐模型训练和真实效果评估。

## 9. 数据要求

V3 至少需要以下对象：

- Organization：企业画像；
- DiagnosticSession：一次联合诊断；
- SelectedEvent：已选事件、顺序和主事件标记；
- QuestionMapping：事件与题项映射；
- Answer：题项回答、阶段和来源；
- RelationCandidate：候选关联及来源事件；
- RelationDecision：用户确认、拒绝或待定；
- RouteCandidate：联合路线及分项得分；
- RouteDependency：路线前置、并行和后置关系；
- ConsultantReview：顾问修订和理由；
- Outcome：项目实施后的经营结果。

## 10. 验收标准

1. 用户可选择 1—3 个事件，不能超过 3 个；
2. 切换诊断视角会清空原事件组合；
3. 必须存在一个主事件才能开始；
4. 重复核心题只展示和回答一次；
5. 交叉题排在单事件题之前并显示支持来源；
6. 核心题完成后只显示初步方向，不显示置信度；
7. 未确认关联事件不调用补充题；
8. 已在核心阶段回答的题不在补充阶段重复出现；
9. 至少一项新增补充回答后才显示联合置信度；
10. 事件关系明确标记为假设而不是已证实因果；
11. 路由能够区分联合优先和候选模块；
12. 320px、768px、1024px 和 1440px 下主要流程可用；
13. 前端 lint 与 production build 通过。

## 11. 成功指标

- 多事件选择率和平均选择数量；
- 合并后题项减少比例；
- 交叉题回答完成率；
- 用户对候选关联的确认、拒绝和修正比例；
- 联合路线被顾问接受或小幅调整的比例；
- 单一联合项目覆盖多个事件的比例；
- 路线拆分与最终项目组合的一致度；
- 项目实施后的多指标联合改善率。

# 面试讲稿：亮点补充与事实边界建议

## 一、当前讲稿最需要修正的表述

1. **“长问卷转化率极低”目前没有真实业务数据支撑。**建议表述为“我识别到一个待验证的漏斗风险：长问卷可能提高进入门槛和中途退出率，因此把官网点击率、隐私说明后进入率和问卷完成率列为后续验证指标”。
2. **当前原型尚未接入 DeepSeek。**已经实现的是浏览器本地规则 Agent；DeepSeek 或其他模型属于下一阶段候选方案。面试时应区分“已实现”“已设计”“待验证”。
3. **当前不是固定生成 8—10 道精准追问。**系统会从企业自述映射最多 6 道候选题，再与经营事件核心题合并、去重。应按真实机制描述。
4. **企业自持密钥保险箱尚未实现。**当前 V3.1 是填写前隐私声明、浏览器运行内存保存、刷新后清除、默认不向和君提交。密钥保险箱是已经完成架构设计的下一阶段能力。
5. **不能承诺“服务方绝对无法查看”或“只有密钥才能打开”。**正式实现客户端加密、密钥管理并完成安全审计后，才可以使用接近零知识的宣传语言。
6. **案例授权不等于模型训练授权。**“脱敏沉淀新案例反哺 AI 模型”应改成：项目结束后另行取得案例授权；如果要使用原始数据训练或优化模型，还必须取得独立训练授权。脱敏也不能替代授权。
7. **不要强调模型的“思维链”。**更适合面试的说法是：要求模型输出结构化标签、候选题、映射理由、缺失信息和不确定性，并通过规则和人工复核控制路由质量。

## 二、讲稿遗漏的核心亮点

### 1. 你不是取消问卷，而是重新定义问卷的角色

这是项目最有价值的产品判断。传统模式把问卷当成最终诊断依据；你的方案将问卷降级为“企业自述之后的定向核验工具”。流程变为：

```text
企业自由描述
→ 可观察经营事件
→ 风险信号与机制假设
→ 候选题项核验
→ 咨询业务路由
```

这样既保留问卷低成本、标准化的优势，又减少固定题库对企业真实语境的压缩。

### 2. 从“主观形容词”切换到“可观察经营事件”

系统不会直接接受“员工执行力差”“团队不好管”作为结论，而是让企业确认：是否存在审批延迟、交付延期、跨部门阻塞、目标持续未完成等真实事件。这使诊断从态度判断转向可被案例、流程和数据核验的事实入口。

### 3. 区分四个结论层级

系统明确区分：

- 风险信号；
- 机制假设；
- 待核验问题；
- 咨询建议。

这体现了 AI 产品经理对模型幻觉和咨询责任边界的控制。系统不把一次自测包装成权威诊断，也不把相关性写成因果关系。

### 4. 映射结果可解释、可修改

Agent 不只推荐题项，还展示命中信号和映射理由；企业可以移除或恢复候选题。AI 是辅助路由者，不是不可质疑的裁判。

### 5. 多事件合并与重复题去重

企业问题通常不是单点问题。系统最多允许选择三个并发经营事件，再把共同核心题合并，交叉题优先显示、重复题只回答一次。这既减少填写负担，也帮助识别多个表象是否可能共享同一组织机制。

### 6. 按回答角色拆分两套入口

业务/职能负责人聚焦执行、团队和业务现场；创始人/总经理聚焦战略、治理、资本配置和高管团队。角色不同，能观察到的事实和适合回答的问题也不同，因此不能混用同一套路由底稿。

### 7. 路由不是直接“推销售”，而是分层连接能力

完整路由是：

```text
问题信号
→ 诊断模块
→ 候选机制
→ 和君咨询/资本/商学
→ 业务单元与服务产品
→ 咨询师候选
```

先匹配组织能力，再匹配个人顾问，避免把复杂企业问题简化为销售人员分单。咨询师匹配还应考虑行业经验、方法、地域、档期和利益冲突，并保留人工复核。

### 8. 案例放在咨询师之前，是一个有意设计的信任顺序

企业首先需要确认“这家机构是否理解并解决过相似机制的问题”，之后才关心“由谁承接”。所以先展示获得授权的相似案例、解决方向、指标口径和非效果承诺，再展示咨询师，比直接推个人名片更符合 B 端采购的信任逻辑。

### 9. 三类授权必须分离

- 咨询沟通授权：允许和君查看企业选择提交的信息；
- 案例展示授权：允许项目结束后按指定字段公开或匿名展示；
- 模型训练授权：允许特定数据用于模型优化。

三者不能捆绑、默认勾选或相互替代。这个设计比单纯“加一个隐私弹窗”更完整。

### 10. 官网嵌入本身是获客策略

产品不是孤立的小程序，而可以直接嵌入和君官网。官网访客本身已有企业管理或咨询意向，点击链路更短，也能把内容营销、初诊、案例证明和咨询授权放进同一漏斗。

### 11. 双商业模式

- 企业自助闭环：单次付费、订阅或私有部署，用于企业内部复盘；
- 诊断导向咨询：初诊后通过案例和服务路由进入正式咨询。

同一诊断底座既能成为软件产品，也能成为咨询获客入口，这是商业化层面的扩展性。

### 12. 反馈闭环不是“自动训练模型”

更成熟的反馈闭环是保存经授权的人工反馈：企业接受或删除了哪些候选事件和题项、顾问是否调整了路由、最终进入了什么项目、结果指标是否改善。先用这些数据评估路由准确性，再决定是否训练模型，而不是把新案例自动喂给 AI。

## 三、推荐的面试主线

```text
原问卷既是诊断底稿又决定咨询路由，主观偏差可能被一路放大
→ 保留问卷，但把它改成企业自述后的定向核验层
→ 用可观察经营事件和角色分流提高问题表达质量
→ 输出风险信号、机制假设和待核验证据，不伪造确定结论
→ 先以授权案例证明组织能力，再路由业务单元和咨询师
→ 通过最小披露及咨询/案例/训练三类授权解决 B 端信任问题
→ 最终形成官网获客与企业自助两种商业模式
```

一句话总结：

> 我不是简单地给问卷加一个 AI 输入框，而是重新设计了咨询入口的信息链、判断链、业务路由链和授权链，让企业更容易表达问题，也让咨询公司获得的线索更可解释、更可承接。

# 企业健康初诊与咨询路由项目——面试讲稿 V2

各位面试官好，今天我想介绍一个我为咨询公司设计的企业健康初诊与咨询路由产品。

这个项目也比较能代表我对 AI 架构师这个角色的理解。我的优势不是先拿着一个最新模型或 Agent 框架去寻找可以包装的场景，而是先理解企业正在做什么、谁在什么情境下遇到了什么问题、现有流程卡在哪里，再判断哪些环节适合使用 AI，哪些环节应该保留规则、人工判断和授权控制。对我来说，AI 架构不是技术名词的组合，而是把合适的 AI 能力放进真实业务流程，并让它产生可验证的业务价值。

这个项目最初给到我的任务，是处理一份企业健康度自测 Excel。原始文件通过固定问卷覆盖战略、组织、人力、营销、财务、数字化等多个管理模块，再根据企业回答生成分析。

我拿到问卷后，最先考虑的不是怎么把它做成网页，也不是先选择什么模型，而是先问了五个产品问题：**这份问卷要用在哪里，谁是使用者，谁来填写，企业为什么愿意填写，以及填写完成以后要做什么。**

因为同一份问卷放在不同的业务场景中，承担的产品职责完全不同。

第一种使用模式，是把它做成企业端自用的闭环服务。企业管理者通过自测识别当前可能存在的战略、组织、团队或经营问题；完成初诊以后，再进入不同的专业 Agent，获得针对性的追问、改善建议和行动指导，并在执行一段时间后重新评估。在这个场景中，产品的核心任务是帮助企业完成“发现问题—理解问题—采取行动—再次评估”的内部闭环。它可以进一步发展成按次付费、订阅或私有部署的小型 SaaS 产品。

第二种使用模式，是由咨询公司把问卷提供给已经产生咨询意向的企业。企业来到咨询公司，通常意味着它已经感受到经营或组织上的问题，但还不一定能把问题说清楚，也不知道应该找哪一类咨询服务。此时，填写问卷并不是流程终点。企业填完之后的下一步动作，应该是进一步确认自己需要解决什么问题、可能采用什么解决方向，以及应该由咨询公司的哪个业务单元和哪位咨询师继续承接。

因此，在咨询公司场景中，问卷承担的职责不只是“测出一个分数”。它首先要帮助企业意识到问题、理解问题；然后还要让企业看见咨询公司能够提供什么方案、是否有解决过类似问题的案例，以及有哪些专业资源可以承接。换句话说，它既是企业初诊工具，也是咨询公司的业务分流入口、能力展示窗口和信任建立环节。

我也因此把项目拆成了两条不同的产品路径：企业自用闭环更强调持续指导和绩效改善；诊断导向咨询更强调问题澄清、案例证明、业务路由和咨询连接。当前原型优先深化的是第二条路径，因为它与咨询公司的获客、转化和服务承接链路结合得更紧密。

在明确了这个使用场景之后，我才继续追问：如果问卷将直接决定企业看见什么问题、又被路由到什么服务，那么这个路由依据本身是否可靠？

也正是在这里，我发现了项目最关键的风险：**如果原始问卷的题项、构念或回答存在偏差，而系统又直接把分数转换成业务推荐，那么这种偏差不会因为数字化而消失，反而可能被算法放大。**尤其是在企业实时事实、访谈材料和顾问复核都比较少的情况下，看起来精确的数据路由可能比传统问卷产生更失真的判断。

所以我继续追问了两个问题：第一，原问卷究竟在测量哪些管理构念，这些构念是否有更权威、更全面的测量底稿；第二，即使量表本身可靠，企业的主观回答能不能直接决定它需要什么咨询服务。

我后来形成的判断是：更权威的量表可以帮助我们校准题库和构念覆盖，但不能简单更换一套问卷就解决路由问题。因为咨询路由既要告诉客户“企业可能哪里有问题”，又要连接咨询公司真实能够提供的解决方案。它需要的不只是测量，还包括经营语境、问题机制、证据完整性和服务承接能力。

因此，我对产品做的第一个重要调整，不是取消问卷，而是重新定义问卷的角色：**问卷不再是唯一诊断底稿，也不直接充当最终结论，而是企业表达问题之后的一组定向核验工具。**

比如企业只说“团队执行力差”，系统不能立刻把它映射成人才培训，因为真正的问题也可能是目标冲突、职责不清、审批过慢或者资源配置不到位。基于这个判断，我才开始重新设计整个诊断入口。

第一步，企业先用自己的语言描述当前难点。输入框会引导企业说明发生了什么、从什么时候开始、影响了什么业务结果，以及已经尝试过哪些措施。当前原型使用浏览器本地规则 Agent 对文字进行解析，把描述转化成增长、交付、协同、人才、数字化等候选经营信号，同时展示命中了哪些信息、为什么推荐相关题项。这里我刻意没有让系统直接给出根因，因为企业自述只能形成风险信号和待核验假设，不能代替正式诊断。

第二步，系统要求企业确认真实发生的经营事件。例如，不直接问“组织健康吗”，而是让企业确认是否出现了跨部门事项反复卡住、交付延期、关键人才持续离开，或者增长投入长期没有回报。企业最多选择三个并发事件，再指定一个当前最希望优先解决的问题。

这样设计有两个价值。第一，它把抽象评价转化成可以被案例、流程和经营数据验证的事实；第二，企业问题通常不是单点问题，多个现象可能共享同一个组织机制，所以系统会合并不同事件的核心题，重复题只回答一次，交叉题优先展示，减少传统长问卷的重复填写。

第三步，我根据回答者角色拆分了两套入口。业务和职能负责人主要回答执行、团队和业务现场问题；创始人和总经理则聚焦战略、治理、资本配置和高管团队。因为不同角色掌握的信息不同，如果让所有人回答同一套题，得到的结果看似标准，实际上可能没有可比性。

完成核心题后，系统不会输出一个覆盖全公司的“健康总分”，而是把结果拆成四个层级：第一是风险信号，第二是可能的机制假设，第三是还需要补充什么材料或真实案例，第四才是可以进一步讨论的咨询方向。对于没有覆盖的模块，系统不会默认企业健康；对于“暂时不清楚”和相互矛盾的回答，也会保留，而不是强行生成确定结论。

接下来是这个产品与咨询业务真正连接的部分。

我把路由设计成了一条分层链路：从企业问题信号和诊断模块出发，先匹配和君咨询、和君资本、和君商学，再匹配相应的业务单元和服务产品，最后才生成咨询师候选。之所以不直接推荐咨询师，是因为复杂的企业问题首先需要确定由什么组织能力承接，而不是简单地把线索分给某个人。正式版本中，咨询师匹配还需要结合真实履历、行业经验、工作方法、地域、档期以及利益冲突，并由人工最终确认。

在结果页，我把相似案例放在咨询师卡片之前。因为对于企业客户来说，第一步不是相信某一位顾问，而是先判断这家咨询机构是否理解并解决过相似机制的问题。案例卡片会展示问题相似度、相似依据、解决方案方向和经授权的结果指标，同时明确案例授权状态和“历史结果不构成当前效果承诺”。建立组织信任之后，再展示适合承接沟通的咨询师，转化路径会更自然。

随后我又补上了一个 B 端产品不能回避的问题，也就是数据隐私。

当前 V3.1 原型在填写前设置了必须滚动阅读的隐私说明：企业填写的信息只保存在当前浏览器运行内存中，刷新或关闭后清除，不会自动提交给咨询师、销售人员，也不会用于模型训练。企业完成初诊以后，可以先在本机查看结果；只有主动打开“申请咨询”，确认联系人和准备提交的字段，才会生成授权预览。

我还把授权拆成了三个互相独立的环节：第一是咨询沟通授权，第二是项目结束后的案例展示授权，第三是模型训练授权。这三种授权不能默认勾选，也不能互相替代。即使企业同意匿名展示案例，也不代表允许把原始诊断内容用于模型训练。

在下一阶段设计中，我规划了企业自持密钥保险箱：诊断数据在浏览器端加密后再保存到云端，企业保留恢复密钥；需要咨询时，由企业在本地解锁，并只把选定字段重新加密给和君，而不是直接把主密钥交给咨询师。不过这部分目前属于已经完成产品和技术方案设计、尚未开发和安全审计的能力，所以我不会把它包装成已经上线的“绝对不可见”。

回到开头的两种使用模式，它们最终形成了两条商业化路径：企业自助闭环可以按单次诊断、订阅或私有部署收费；诊断导向咨询则可以直接嵌入和君官网，让本身已经对企业管理或咨询感兴趣的访客，用更短的路径完成问题表达、初诊、案例验证和咨询授权。这样，同一套诊断与路由底座既可以成为小型 SaaS 产品，也可以成为咨询公司的智能获客和业务分流入口。

目前我已经完成了 V3.1 可交互网页原型，包括隐私前置、自由问题输入、本地问题映射、角色分流、经营事件选择、核心题合并、初诊分析、业务路由、模拟案例卡片、虚拟咨询师推荐和授权预览。我也完成了硅基流动模型的服务端接入层和资深咨询师 Skill；由于当前真实推理接口存在超时，原型仍保留本地规则降级。当前案例、咨询师、业务单元名称和结果数据都明确标记为演示内容，云端密钥保险箱、真实案例库和正式咨询师目录还没有接入。

如果继续推进，我不会第一步就训练模型，而是先验证三个问题：企业是否愿意完成这条诊断路径；企业和顾问是否认可系统推荐的事件、题项和业务路由；相似案例是否真的能够提升咨询授权率。只有积累了企业对映射的修改、顾问对路由的复核，以及最终项目去向等经授权反馈后，才有基础评估是否需要模型化。

所以，这个项目对我来说并不是简单地给问卷增加一个 AI 输入框，也不是为了证明我使用了多么新的模型，而是重新设计了咨询入口的信息链、判断链、信任链、业务路由链和授权链。AI在这里不是替代咨询师直接下结论，而是帮助企业把问题说清楚，帮助咨询公司把线索路由到真正能够承接的能力，同时保留企业对数据和最终决策的控制权。

我希望体现的能力是：面对一个已经存在的业务工具，我能够先还原它的使用场景和后续动作，再把业务问题拆成可以由模型、规则、Agent和人工共同承担的任务，最后把这些能力重新装回业务流程中。技术对我来说不是展示目的，而是完成业务目标的组件；我真正负责的是这些组件应该在哪里出现、为什么出现，以及如何证明它确实改善了原来的工作方式。

以上就是我的项目介绍，谢谢各位面试官。

## 一句话收尾版

> 我的优势不是从新技术出发寻找使用场景，而是从真实业务场景出发，为模型、规则、Agent和人工找到各自应该承担的位置，并把它们组合成能够落地的业务系统。

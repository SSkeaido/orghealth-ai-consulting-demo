# OrgHealth V2 PRD：企业经营事件与咨询路由工作台

> 版本：V2.0  
> 日期：2026-07-22  
> 产品阶段：交互原型 → 可用 MVP  
> 适用业务：咨询获客入口、单次付费诊断、企业持续经营陪跑

## 1. 版本摘要

OrgHealth V2 不再把“企业健康度测评”作为用户看到的主体，也不要求企业完成固定题目或上传开放式资料。产品以企业当前发生的经营事件为入口，让企业在工作台中持续维护关键事实、指标变化、管理动作和验证状态；系统在后台调用诊断框架，形成问题假设、因果关系和下一步咨询路线。

V2 的核心链路：

```text
经营事件 → 经营事实 → 问题假设 → 原因验证 → 咨询路由 → 子 Agent / 真人顾问 → 改善追踪
```

V1 的 56 项、18 模块和五大维度继续存在，但退为后台诊断知识库，不再以固定问卷呈现。

## 2. 产品定位

### 2.1 一句话定位

以企业经营事件为入口，帮助企业找到最值得优先解决的问题，并路由到合适咨询服务与专项 Agent 的智能咨询工作台。

### 2.2 两种业务模式

#### A. 咨询公司的业务入口

- 企业通过链接进入轻量工作台；
- 选择当前经营事件并补充关键事实；
- 路由 Agent 输出候选问题和咨询路径；
- 咨询顾问复核后发起专项咨询项目；
- 后续子 Agent 继承已有事实和问题图谱。

#### B. 企业单次或持续使用

- 单次付费：完成一次经营事件分析和咨询路线建议；
- 周期复盘：按月或季度更新事实，观察问题变化；
- 企业专属版：保留历史事件、组织知识和改善记录；
- 轻量形态：网页、H5、小程序或单次访问码，不急于建设完整 SaaS。

## 3. V2 设计原则

1. **事件先于测评**：从“现在发生了什么”开始，而不是从企业全量健康检查开始。
2. **事实先于结论**：用户维护事实，系统生成候选假设；未经验证不称为结论。
3. **后台测量**：56 项诊断框架、量表和成熟度规则仅作为后台推理依据。
4. **渐进建模**：每增加一条事实，问题图谱和路由建议随之更新。
5. **推荐可解释**：每个咨询推荐必须说明触发事实、可能根因、置信度和缺失信息。
6. **顾问可接管**：企业自助阶段输出初步路径，正式诊断和方案需顾问确认。
7. **不以成交为唯一优化目标**：优先优化企业改善价值、适配度和实施成功概率。

## 4. 目标用户与角色

| 角色 | 核心任务 | 可见内容 |
|---|---|---|
| 企业负责人 | 描述经营事件、确认事实、选择咨询方向 | 企业工作台、初步路径、行动任务 |
| 企业项目联系人 | 更新指标、事实和改善进展 | 事实台账、任务和状态 |
| 咨询顾问 | 复核假设、调整因果关系、发起咨询项目 | 全部事实、问题图谱、路由解释 |
| 专项子 Agent | 承接特定咨询方向 | 与该项目相关的事实、假设和任务 |
| 咨询管理者 | 配置服务产品和分流规则 | 咨询服务库、路由规则、转化数据 |

## 5. 经营事件模型

### 5.1 MVP 事件类型

- 收入增长停滞或下降；
- 利润或现金流承压；
- 客户流失或获客困难；
- 项目延期或交付质量下降；
- 跨部门协同和决策效率下降；
- 人才流失或关键岗位缺口；
- 数字化投入效果不明显；
- 融资、并购或上市准备；
- 新业务或创新项目推进困难；
- 企业主动进行周期经营体检。

### 5.2 事件对象

每个事件包含：事件类型、发生时间、影响范围、严重程度、当前阶段、企业目标、已知事实、候选问题、待验证事项、路由结果和改善状态。

### 5.3 事件状态

```text
已创建 → 补充事实 → 形成假设 → 原因验证 → 路由就绪 → 咨询进行中 → 改善追踪 → 已复盘
```

## 6. 信息输入方式

V2 不设置固定长问卷，也不要求开放式上传。信息通过工作台中的经营对象维护。

### 6.1 项目创建时的最小输入

- 企业名称：短文本；
- 所属行业：选择；
- 企业规模：区间选择；
- 当前经营事件：事件卡单选；
- 事件发生时间：月份选择；
- 希望改善的时间：1/3/6/12 个月。

### 6.2 事实台账

一条事实不是一道题，而是一条可以持续更新的经营记录：

- 事实类型：指标、事件、管理动作、人员变化、客户反馈；
- 指标或事实名称：从字典选择；
- 当前值与对比值；
- 时间范围；
- 信息来源：企业自述、系统数据、顾问确认、第三方；
- 验证状态：未验证、部分验证、已验证、有冲突；
- 关联经营事件。

### 6.3 管理动作

记录企业已经采取的动作、负责人、开始时间、预期结果和当前效果，避免系统重复推荐已经实施的方案。

## 7. 核心工作台

### 7.1 经营驾驶舱

展示当前经营事件、事实完整度、待验证假设、路由就绪度和下一步动作。不得在证据不足时显示“健康度低”或正式风险结论。

### 7.2 事实台账

- 按事件、时间、来源和验证状态查看；
- 快速新增指标、事件或管理动作；
- 显示企业自述与顾问确认的区别；
- 修改关键事实后触发问题图谱和推荐重新计算；
- 保留变更历史。

### 7.3 问题图谱

将事实、候选问题、可能根因和经营结果组织为可解释关系图：

```text
收入下降
├─ 新客户减少
│  ├─ 线索不足
│  └─ 转化率下降
├─ 老客户流失
│  ├─ 交付延期
│  └─ 产品价值下降
└─ 客单价下降
```

每个节点标记：类型、置信度、支持事实、反向事实、验证状态和前置依赖。

### 7.4 咨询路由

输出 Top 3 路径，而不是唯一答案：

- 咨询方向和专项 Agent；
- 推荐分与置信度；
- 触发事实；
- 推断链路；
- 缺失信息；
- 前置条件；
- 建议周期和交付物；
- 是否需要真人顾问；
- 接受、调整、拒绝及原因。

### 7.5 执行追踪

咨询路线被接受后创建改善项目，包含目标、关键结果、任务、负责人、时间、指标和复盘记录。改善结果反向进入推荐反馈数据。

## 8. 路由 Agent

### 8.1 输入

- 企业画像；
- 当前经营事件；
- 已验证和未验证事实；
- 问题图谱；
- 企业目标与资源约束；
- 已采取管理动作；
- 咨询服务库与适用条件；
- 历史顾问反馈和改善结果。

### 8.2 候选生成

先由业务规则和问题依赖图生成候选咨询方向，避免模型从所有服务中自由猜测。

### 8.3 排序模型

MVP 使用可解释加权排序：

```text
推荐分 =
问题严重程度 × 20%
+ 业务影响 × 25%
+ 战略相关性 × 15%
+ 改善可行性 × 15%
+ 证据可信度 × 15%
+ 服务适配度 × 10%
- 前置条件缺失惩罚
```

不得将所有因素直接相乘；单个低分不应无解释地把总分压到零。

### 8.4 后续优化

- 规则推荐；
- 加权排序；
- 相似案例检索；
- Learning to Rank；
- 基于改善效果的长期策略优化。

主要优化目标：企业改善价值 × 服务适配度 × 实施成功概率。成交概率仅作为商业辅助指标。

### 8.5 人工反馈

顾问可接受、调整或拒绝推荐并填写原因；企业可反馈是否理解、是否愿意继续和未选择原因；项目结束后记录改善效果。这些数据用于优化路由，而不是直接使用企业原始敏感数据训练通用模型。

## 9. 专项子 Agent

MVP 优先提供：

1. 战略与经营 Agent；
2. 组织与人才 Agent；
3. 财务与经营效率 Agent。

后续增加市场销售、数字化转型、创新管理和资本规划 Agent。

子 Agent 必须继承：企业画像、当前事件、事实台账、问题图谱、路由理由和顾问修订，不让企业重复输入。

## 10. 页面信息架构

### 10.1 入口页

企业名称、行业、规模和经营事件选择，一个主按钮“建立经营事件”。不出现健康度题目、资料上传或雷达图。

### 10.2 工作台导航

- 经营驾驶舱；
- 事实台账；
- 问题图谱；
- 咨询路由；
- 执行追踪。

### 10.3 企业端与顾问端差异

企业端看事实、候选问题和咨询路线；顾问端额外看到后台诊断映射、证据置信、路由权重、冲突和复核工具。

## 11. 数据模型

- Organization：企业画像；
- BusinessEvent：经营事件；
- BusinessFact：经营事实及来源；
- MetricDefinition：指标字典与口径；
- ManagementAction：已采取的管理动作；
- ProblemNode：问题、原因和结果节点；
- ProblemEdge：因果、相关、前置或反向关系；
- DiagnosticMapping：问题节点与后台诊断构念映射；
- ConsultingService：咨询产品、适用条件和交付物；
- RouteCandidate：候选咨询路线和分项得分；
- RouteDecision：接受、调整、拒绝和原因；
- AgentHandoff：子 Agent 接力上下文；
- ImprovementProject：改善项目；
- OutcomeMetric：实施结果；
- FeedbackRecord：顾问、企业和结果反馈。

## 12. AI 边界

- 可以基于事实生成候选问题和解释；
- 可以推荐下一项最有信息价值的事实；
- 可以生成咨询路线候选和推断链；
- 不得把企业选择的经营事件当成已证实根因；
- 不得把未验证事实写成正式结论；
- 不得仅按最低健康度分数机械路由；
- 不得为了提高成交率故意放大问题；
- 所有正式建议必须显示事实依据和置信限制。

## 13. MVP 范围

### 必须实现

- 事件式项目创建；
- 驾驶舱；
- 事实台账和结构化新增；
- 问题图谱展示；
- 三条咨询路线及解释；
- 推荐接受/拒绝反馈；
- 执行追踪框架；
- V1 与 V2 可独立访问。

### 暂不实现

- 真实模型训练；
- 自动读取企业系统；
- 复杂多租户和计费；
- 全行业咨询产品库；
- 无人工复核的正式咨询报告。

## 14. 验收标准

1. 新建项目不出现固定问卷和开放式文件上传；
2. 用户可在 2 分钟内建立一个经营事件；
3. 工作台以事实、问题和路线为主要对象；
4. 所有候选问题明确标记“待验证”；
5. 路由推荐至少展示两项触发事实和一项缺失信息；
6. 用户修改关键事实后路由就绪度发生变化；
7. 推荐结果支持接受、调整和拒绝；
8. 接受路线后可创建执行追踪项目；
9. V1 前端不被删除，可通过独立入口访问；
10. 前端 lint 和 production build 通过。

## 15. 成功指标

- 事件创建完成率；
- 从入口到路由就绪的时间；
- 路由推荐被顾问接受或小幅调整的比例；
- 每个推荐的可解释事实覆盖率；
- 企业进入专项咨询的转化率；
- 咨询项目改善指标达成率；
- 顾问调整推荐的原因分布；
- 周期复盘企业的留存率。

## 16. V1 到 V2 的迁移

- 保留 56 项、18 模块和量表研究作为后台知识资产；
- 将 V1 的上传、成熟度评分和报告页面保留为顾问深度诊断工具；
- V2 企业端不直接展示统一健康度总分；
- V2 问题节点可映射到 V1 模块，用于顾问接管后的深度诊断；
- 已有 V1 演示数据不得自动进入 V2 的正式事实台账。

> 文档版本：V4.0

> 更新日期：2026-07-22  
> 适用项目：OrgHealth V3/V3.1 后续迭代

> 文档状态：待产品、咨询业务、信息安全与法务共同评审

## 1. 产品摘要

### 1.1 一句话定位

一个由企业自主控制数据的咨询入口：从企业正在发生的经营问题出发，通过问题映射、题项核验、诊断分析、相似案例、和君业务与咨询师匹配，帮助企业决定是否发起正式咨询。

### 1.2 核心价值

对企业客户：

- 不要求用户先理解咨询产品，而是从自身正在发生的问题进入；
- 将自由描述转化为可解释、可修改的候选题项；
- 告诉企业“哪里可能有问题、为什么这样判断、还缺什么证据”；
- 展示与当前问题相似且已获授权的案例、解决方案和成效；
- 推荐可能适配的和君业务与咨询师；
- 企业默认掌握数据控制权，未主动授权前和君不可查看。

对和君：

- 将官网访问流量转化为结构化咨询线索；
- 在不提前获取企业敏感信息的前提下建立信任；
- 将客户问题路由到适合的业务单元、服务产品和咨询师；
- 通过经授权的案例库展示实际交付能力；
- 积累经企业另行授权的案例、路由和项目结果数据。

### 1.3 产品边界

本产品是咨询入口和业务路由工具，不是自动出具权威企业诊断的系统。系统输出分为四层：

1. **风险信号**：由企业自述或题项回答触发；
2. **机制假设**：对可能原因的候选解释；
3. **待核验问题**：需要材料、访谈或顾问确认；
4. **咨询建议**：适合进一步讨论的服务方向。

任何自动输出均不得直接表述为已确认根因、审计结论、法律意见或投资建议。

## 2. 背景与机会

传统健康度自测依赖固定问卷。问卷成本低、易使用，但存在三类问题：

- 用户只能在既定题目中表达，真实经营语境容易丢失；
- 企业主观回答可能受角色、认知与社会赞许偏差影响；
- 问卷得分与咨询业务之间缺少透明、可解释的路由。

本产品不取消问卷，而是改变问卷的角色：问卷不再是唯一诊断底稿，而是企业自述之后的一组定向核验工具。系统先理解用户描述的经营问题，再提出候选事件和题项，由企业确认后进入分析。

和君集团简介披露的业务格局为“咨询、资本、商学”一体两翼，并具备战略、组织流程、人力文化、品牌营销、内控风险、精益供应链、数字化与 AI、资本运作和企业培训等服务能力。产品需要把企业问题与这些服务能力建立可维护的映射，而不是在结果页放置统一销售入口。

## 3. 商业模式

### 3.1 模式 A：企业自助使用闭环

企业购买一次或一定周期的诊断使用权，自主完成问题描述、事件选择、题项核验和结果查看。和君默认不获取诊断内容。

适用场景：

- 企业内部经营复盘；
- 管理层会议准备；
- 部门专项自检；
- 定期组织健康跟踪。

可能收费方式：单次诊断、企业账号订阅、专项模块包。

### 3.2 模式 B：诊断导向咨询

企业完成初诊后，通过相似案例、服务能力与咨询师推荐判断是否发起咨询。只有企业主动授权指定数据后，和君才能接收信息并跟进。

主要转化链：

```text
官网访问
  → 了解诊断方法
  → 隐私与数据控制说明
  → 企业自助初诊
  → 初步问题与证据边界
  → 相似授权案例
  → 和君业务能力
  → 咨询师推荐
  → 企业主动授权提交
  → 顾问沟通与正式咨询
```

### 3.3 两种模式的共同原则

- 完成诊断不自动生成销售线索；
- 企业不申请咨询时，和君人员不可读取其诊断；
- 咨询申请授权与案例公开授权相互独立；
- 不以隐藏诊断结果强迫用户留下联系方式；
- 案例展示用于证明适配经验，不能承诺相同结果。

## 4. 用户与角色

| 角色 | 主要目标 | 主要担忧 | 核心权限 |
|---|---|---|---|
| 企业业务/职能负责人 | 判断执行、团队和业务现场问题 | 信息泄露、被销售打扰、结论失真 | 创建诊断、选择提交范围 |
| 创始人/CEO/总经理 | 判断战略、组织、资本与阶段问题 | 重大经营信息暴露、误判根因 | 使用 CEO 事件库、授权顾问访问 |
| 企业数据授权人 | 控制哪些信息可以离开企业保险箱 | 授权范围失控、无法撤回 | 解锁、授权、撤回、设置期限 |
| 和君线索协调人 | 将授权线索分配给适合团队 | 信息不足、错误分流 | 仅查看已授权字段 |
| 和君咨询师 | 理解问题并准备初谈 | 推荐理由不透明、案例冲突 | 查看获授权诊断、补充顾问判断 |
| 案例库管理员 | 管理可公开案例与授权范围 | 越权公开、指标口径不一致 | 审核授权、脱敏、发布和下架 |
| 系统管理员 | 保证服务可用与安全 | 越权解密、审计缺失 | 管理密文和系统配置，不默认拥有明文权限 |

## 5. 产品设计原则

1. **从企业问题出发**：先理解用户处境，再选择工具和服务。
2. **问卷用于核验**：自由文本不直接生成结论，题项不脱离构念和适用边界。
3. **解释优先**：任何事件、题项、服务、案例和咨询师推荐必须显示依据。
4. **企业可修改**：Agent 的候选映射必须可移除、恢复或拒绝。
5. **共同机制优先**：多事件场景先寻找共享机制，再决定是否拆分专项。
6. **默认私密**：完成评测不等于提交数据。
7. **授权最小化**：只提交完成特定目的所需的字段和时限。
8. **案例必须有授权**：无有效授权状态的案例不得出现在客户侧。
9. **结果不夸大**：案例结果不构成对当前企业的效果承诺。
10. **人工兜底**：正式咨询路由由顾问或线索协调人确认。

## 6. 版本范围

### 6.1 当前原型已实现

- 官网式诊断方法介绍与进入诊断流程；
- 企业名称、行业、规模和答题角色采集；
- 企业当前难点的自由文本输入；
- 本地规则版问题映射 Agent；
- 自述信号到经营事件、构念模块和候选题项的映射；
- 事件选择、主事件指定、题项合并和交叉题优先；
- 问卷回答分析、覆盖范围、风险信号和一致性检查；
- 企业问题到和君演示业务单元的路由；
- 模拟案例相似度、解决方案、模拟成效与评价卡片；
- 虚拟咨询师匹配卡片；
- 填写前隐私声明；
- 咨询申请与案例授权分离的交互；
- 当前无后端提交，咨询授权只生成预览；
- 响应式前端和生产构建。

### 6.2 P0：隐私型 MVP 必须建设

- 企业自持密钥保险箱；
- 浏览器端加密后上传密文；
- 诊断会话恢复、自动过期和主动删除；
- 企业授权后按字段重新加密给和君；
- 和君机构公钥、密钥轮换和访问审计；
- 正式咨询申请接口和顾问端只读视图；
- 真实业务目录、咨询产品目录和咨询师资料；
- 案例授权状态、脱敏级别和发布审核；
- 隐私政策、授权文本与法务评审。

### 6.3 P1：路由质量与运营

- 外部模型或企业本地模型接入；
- 顾问对问题、题项、业务和案例映射的反馈；
- 路由版本管理和灰度测试；
- 真实项目结果回填；
- 企业授权撤回、访问期限延长和导出；
- 线索协调、顾问排期与利益冲突检查。

### 6.4 P2：企业持续使用

- 多周期诊断对比；
- 企业内部多角色交叉回答；
- 管理层共识与分歧分析；
- 企业自建题库与服务路由；
- 单次付费、订阅和企业私有部署。

## 7. 端到端用户流程

```text
了解诊断方法
  → 阅读隐私声明
  → 创建企业保险箱（P0）
  → 填写企业背景与经营难点
  → 问题映射 Agent 提取信号
  → 企业确认候选经营事件
  → 企业确认或移除候选题项
  → 完成核心题
  → 查看覆盖范围、风险信号和机制假设
  → 查看相似且已授权案例
  → 查看和君业务单元与咨询师推荐
  → 选择仅本机/保险箱保存，或申请咨询
  → 选择提交字段、期限和联系人
  → 企业本地解锁并授权
  → 指定内容重新加密给和君
  → 和君顾问查看获授权信息并沟通
  → 项目结束后另行询问案例授权
```

## 8. 页面与功能需求

### 8.1 官网入口与诊断方法页

目标：让有咨询兴趣的官网访客理解诊断方法和数据边界，再决定是否进入。

要求：

- 首屏不直接展示复杂问卷；
- “先了解诊断方法”进入连续的方法说明；
- 方法说明结束后才出现“进入企业健康初诊”；
- 说明诊断只产生初步信号，不替代访谈、材料和顾问核验；
- 提前说明数据默认不会提交给和君。

### 8.2 隐私声明与保险箱创建

进入填写前必须展示：

- 数据在哪里处理；
- 是否上传；
- 是否用于模型训练；
- 谁能够查看；
- 数据何时删除；
- 忘记密钥会发生什么；
- 申请咨询时如何授权。

当前原型文案：数据只存在当前浏览器运行内存，刷新或关闭后清除。

P0 正式流程：

1. 系统生成高强度恢复密钥，不允许仅使用简单自定义短密码；
2. 企业可设置口令，用密钥派生函数生成密钥加密密钥；
3. 随机生成数据密钥，加密诊断数据；
4. 服务器只保存密文、被包装的数据密钥和必要元数据；
5. 页面展示保险箱 ID、恢复密钥保存提醒和自动过期时间；
6. 默认不得提供和君后台“万能解密”能力；
7. 恢复机制不能绕过企业的数据控制承诺。

### 8.3 企业背景与自由问题输入

字段：

- 企业名称；
- 行业；
- 规模；
- 回答角色；
- 当前最想解决的经营难点，自由文本，最多 800 字。

自由文本提示应引导用户描述：

- 发生了什么；
- 何时开始；
- 影响了什么业务结果；
- 涉及哪些团队或角色；
- 已经尝试过什么。

不得要求用户输入身份证号、个人敏感信息、客户名单、合同原文或其他完成初诊不必要的数据。

### 8.4 问题映射 Agent

输入：企业自由描述、行业、规模、回答角色。

输出：

- 经营信号；
- 命中词或语义依据；
- 机制解释；
- 候选经营事件；
- 候选题项；
- 匹配置信度；
- 信息不足时的补充追问；
- 诚实边界。

映射规则：

```text
企业描述
  → 可观察现象
  → 经营信号
  → 机制假设
  → 构念/模块
  → 候选题项
  → 企业确认
```

约束：

- 低信息量描述不得强行映射；
- 不得把“员工不积极”等表述直接归因为个人态度；
- 应同时检查目标、能力、资源、激励、权责、流程和管理支持；
- 每个候选题项必须提供映射理由；
- 企业可以移除或恢复题项；
- 映射本身不进入成熟度得分。

### 8.5 经营事件确认

- 专项视角最多选择 3 个事件；
- CEO 视角最多选择 3 个事件；
- 两个事件库不得混选；
- Agent 推荐事件必须明确标记为建议；
- 用户最终确认哪些事件正在真实发生；
- 需要指定一个当前优先事件；
- 切换回答角色时清空不兼容的事件选择。

### 8.6 核心题与题项来源

题项来源分为：

- 所选事件核心题；
- 多事件交叉题；
- 企业自述映射题；
- 后续关联补充题。

要求：

- 重复题只回答一次；
- 交叉题优先显示；
- 每道题显示来源；
- 企业自述映射题不得伪装成事件核心题；
- “暂时不清楚”是有效选项，不得强迫猜测；
- 问卷得分只基于实际回答，不将未覆盖模块视为健康。

### 8.7 初诊分析

输出：

- 本次覆盖模块；
- 各模块局部成熟度；
- 低成熟度风险信号；
- 当前相对稳定项；
- 经营事件与题项回答的一致性；
- 候选机制方向；
- 信息质量状态；
- 下一步需要的案例、制度、流程或经营数据。

禁止：

- 输出未经测量的企业综合健康分；
- 把相关关系写成因果；
- 用单一阈值自动出售咨询服务；
- 隐藏“不知道”和矛盾回答。

### 8.8 和君业务路由

路由层级：

```text
问题信号
  → 诊断模块
  → 候选机制
  → 和君三大业务
  → 建议承接业务单元
  → 服务产品
```

当前演示业务：

- 和君咨询：战略与集团管控；
- 和君咨询：组织、流程与人效；
- 和君咨询：品牌、营销与客户增长；
- 和君咨询：精益运营与供应链；
- 和君咨询：数字化转型与 AI 应用；
- 和君咨询：内控、风险与治理；
- 和君资本：资本运作与融资；
- 和君商学：企业培训与人才培养。

注意：当前业务单元名称为原型演示，不代表和君正式组织编制。P0 必须接入经过业务负责人确认的正式目录。

每个路由必须展示：

- 推荐业务；
- 适用服务；
- 支持信号和题项；
- 不确定性；
- 需要人工确认的内容。

### 8.9 相似案例卡片

案例展示顺序必须在咨询师推荐之前，因为案例用于证明组织能力，咨询师用于承接具体沟通。

案例卡片字段：

- 案例标题；
- 行业和规模区间；
- 授权状态；
- 匿名/实名状态；
- 问题相似度；
- 相似依据；
- 采用的解决方案；
- 结果指标及统计口径；
- 客户评价及授权范围；
- 结果不构成效果承诺的声明。

相似度由以下维度组成：

- 问题信号相似度；
- 构念和题项相似度；
- 业务单元和解决方案相似度；
- 行业与企业阶段相似度；
- 证据完整性；
- 案例授权可展示范围。

禁止将行业相同直接解释为问题相同。

#### 案例授权状态

| 状态 | 客户侧可展示 |
|---|---|
| 未申请授权 | 否 |
| 等待企业确认 | 否 |
| 仅内部复盘 | 否 |
| 已授权匿名展示 | 仅展示授权字段 |
| 已授权实名展示 | 仅展示授权字段 |
| 授权已到期或撤回 | 立即下架 |

当前原型案例均为模拟数据，必须显示“模拟已授权展示 · 非真实客户案例”。

### 8.10 咨询师匹配

咨询师匹配输入：

- 推荐业务单元；
- 服务方向；
- 行业经验；
- 企业阶段；
- 问题机制；
- 工作方法；
- 地域和语言；
- 可用档期；
- 当前项目冲突；
- 利益冲突。

咨询师卡片字段：

- 姓名和职级；
- 常驻地；
- 从业年限；
- 擅长议题；
- 行业经验；
- 建议工作方法；
- 匹配理由；
- 可用档期；
- 资料真实性标识。

当前姓名、履历和档期均为虚拟演示数据。正式版不得在未验证履历、档期和冲突信息时标记为“已匹配”。

### 8.11 企业主动咨询授权

默认状态：和君不可见。

企业点击“申请咨询”后，系统必须显示即将提交的具体范围，例如：

- 企业名称；
- 行业和规模；
- 联系人；
- 企业原始问题描述；
- 已选经营事件；
- 题项回答；
- 初诊分析；
- 企业选择的业务和咨询师偏好。

企业可逐项选择，系统记录：

- 授权人；
- 授权时间；
- 授权文本版本；
- 提交字段；
- 使用目的；
- 接收方；
- 有效期；
- 撤回状态；
- 访问记录。

提交成功后，咨询师也只能查看授权范围内的数据。

### 8.12 案例库二次授权

咨询申请授权不包含案例库授权。项目结束后，如果和君希望将项目收入案例库，必须发起独立流程：

1. 向企业展示拟入库内容；
2. 选择实名、匿名或仅内部；
3. 逐项确认可展示的问题、方案、指标和评价；
4. 标明展示渠道与有效期；
5. 提供撤回方式；
6. 记录授权版本；
7. 案例管理员审核后发布。

不允许使用默认勾选、捆绑授权或“不同意则无法获得咨询结果”等方式获取案例授权。

## 9. 企业自持密钥保险箱

### 9.1 目标

实现“云端可保存、平台默认不可读、企业主动授权后才可分享指定内容”。

### 9.2 推荐架构

```text
诊断数据 JSON
  → 浏览器生成随机数据密钥 DEK
  → 使用认证加密生成密文
  → 企业恢复密钥/口令派生 KEK
  → KEK 包装 DEK
  → 云端保存：密文 + 包装后的 DEK + 盐 + nonce + 版本 + 过期时间

申请咨询时：
企业本地解锁
  → 选择提交字段
  → 生成最小披露数据包
  → 使用和君机构公钥重新加密
  → 上传授权包与授权记录
```

### 9.3 密钥要求

- 不允许将企业自定义的简单短密码直接作为数据加密密钥；
- 系统生成高熵恢复密钥；
- 口令仅用于派生密钥加密密钥；
- 盐必须随机且每个保险箱唯一；
- 使用认证加密，防止密文被篡改而无法发现；
- 主恢复密钥不得上传；
- 和君不得持有可解锁所有保险箱的通用密钥；
- 企业丢失密钥时，必须明确可能无法恢复数据；
- 如果提供恢复机制，必须说明它会削弱何种“平台不可读”承诺。

### 9.4 解锁与分享

不建议企业把主密钥直接告诉咨询师。正确流程是企业在本地解锁，并将选定内容重新加密给和君机构公钥。

分享授权应支持：

- 字段级范围；
- 接收业务单元；
- 指定咨询师或协调角色；
- 访问次数或有效期；
- 撤回；
- 下载限制；
- 完整审计记录。

### 9.5 数据生命周期

| 阶段 | 数据状态 | 和君可见性 |
|---|---|---|
| 填写中 | 浏览器内存明文 | 不可见 |
| 已封存 | 云端密文 | 不可见 |
| 企业本地解锁 | 浏览器短时明文 | 不可见 |
| 已授权提交 | 指定字段的机构公钥密文 | 授权角色可见 |
| 授权到期 | 禁止继续访问 | 不可见 |
| 企业删除 | 密文和包装密钥删除 | 不可见 |

### 9.6 模型处理边界

如果自由文本发送给云端模型处理，模型服务在推理过程中通常需要处理明文，因此必须独立披露：

- 模型供应商；
- 数据发送范围；
- 保存期限；
- 是否用于训练；
- 所在地域；
- 删除方式。

可选路径：

1. 浏览器本地规则 Agent；
2. 浏览器端或企业本地模型；
3. 企业明确授权的云端模型；
4. 私有部署模型。

“咨询师不可见”和“模型不可见”是两个不同的权限，必须分别表达。

## 10. Agent 架构

### 10.1 Agent 划分

| Agent | 输入 | 输出 | 是否可直接生成企业结论 |
|---|---|---|---|
| 问题解析 Agent | 企业自由文本 | 信号、机制假设、追问 | 否 |
| 题项映射 Agent | 信号、角色、题库 | 候选题项及理由 | 否 |
| 诊断分析 Agent | 题项回答、覆盖范围 | 风险信号、一致性、机制方向 | 否 |
| 服务路由 Agent | 诊断结果、和君目录 | 业务单元与服务候选 | 否 |
| 案例匹配 Agent | 问题、方案、行业、授权案例 | 相似案例及相似度 | 否 |
| 咨询师匹配 Agent | 业务、案例、顾问目录、档期 | 顾问候选与理由 | 否 |
| 授权 Agent | 企业选择、保险箱状态 | 最小披露数据包、授权记录 | 不适用 |

### 10.2 统一输出协议

每个推荐对象至少包含：

```json
{
  "candidate_id": "string",
  "candidate_type": "event|question|service|case|consultant",
  "score": 0,
  "confidence": "low|medium|high",
  "reasons": [],
  "supporting_evidence": [],
  "missing_information": [],
  "limitations": [],
  "rule_version": "string",
  "model_version": "string|null"
}
```

### 10.3 人工反馈

正式版必须保存：

- 企业接受、移除或修改了哪些映射；
- 顾问接受、调整或拒绝了哪些路由；
- 最终实际进入的咨询项目；
- 结果是否改善及指标口径；
- 推荐偏差原因。

未经企业许可，不得将企业原始文本用于模型训练。即使获得案例展示授权，也不能自动视为训练授权。

## 11. 数据模型

### 11.1 核心对象

- `OrganizationProfile`：企业画像；
- `DiagnosticVault`：保险箱、密文版本、过期状态；
- `DiagnosticSession`：一次诊断会话；
- `ProblemStatement`：企业原始描述；
- `ProblemSignal`：Agent 提取信号；
- `EventCandidate`：经营事件候选；
- `QuestionMapping`：题项映射与来源；
- `Answer`：企业回答；
- `DiagnosisAnalysis`：分析结果；
- `ServiceDirectory`：和君业务与服务目录；
- `ServiceRoute`：服务路由；
- `CaseProfile`：案例内容；
- `CaseAuthorization`：案例授权范围与状态；
- `CaseMatch`：案例相似度；
- `ConsultantProfile`：咨询师资料；
- `ConsultantMatch`：咨询师匹配；
- `DisclosureGrant`：企业向和君的数据授权；
- `AccessAudit`：解锁、查看、下载和撤回记录；
- `ConsultingEngagement`：正式咨询项目；
- `Outcome`：项目结果与口径。

### 11.2 敏感数据分级

| 级别 | 示例 | 默认策略 |
|---|---|---|
| L0 公开 | 和君业务目录、公开方法 | 可公开 |
| L1 企业基础 | 行业、规模 | 默认加密，不主动提交 |
| L2 企业诊断 | 问题描述、回答、结果 | 企业保险箱，授权后最小披露 |
| L3 经营证据 | 财务、客户、合同、人员材料 | 默认不进入初诊；专项授权 |
| L4 个人敏感 | 身份、健康、薪酬等 | 初诊原则上禁止采集 |

## 12. 后端与接口需求

P0 最小接口：

- `POST /vaults`：创建保险箱，只接收加密包；
- `PUT /vaults/{id}/ciphertext`：更新密文版本；
- `GET /vaults/{id}/ciphertext`：凭保险箱 ID 获取密文；
- `DELETE /vaults/{id}`：删除保险箱；
- `POST /grants`：创建咨询授权包；
- `POST /grants/{id}/revoke`：撤回授权；
- `GET /consultant/grants/{id}`：授权角色读取加密包；
- `POST /access-audits`：记录查看、下载和失败访问；
- `GET /service-directory`：获取业务与服务目录；
- `GET /consultants`：获取可推荐咨询师；
- `GET /cases`：只返回可公开且授权有效的案例。

服务器不得接收企业主恢复密钥。日志不得记录明文诊断、口令、恢复密钥或完整授权数据包。

## 13. 非功能需求

### 13.1 安全

- 全站 HTTPS；
- 浏览器端认证加密；
- 严格内容安全策略，降低脚本注入风险；
- 依赖完整性和供应链扫描；
- 机构密钥轮换；
- 最小权限；
- 访问审计；
- 速率限制和暴力破解防护；
- 保险箱自动过期；
- 安全测试和第三方审计后才可宣称“零知识”或同等级安全能力。

### 13.2 隐私

- 默认不收集；
- 目的限定；
- 数据最小化；
- 授权可撤回；
- 案例授权与训练授权分离；
- 正式上线前由法务根据实际部署地域和客户类型评审。

### 13.3 可用性

- 320px、768px、1024px 和 1440px 可用；
- 关键流程可通过键盘完成；
- 所有表单有可见标签和错误提示；
- 密钥创建、保存和丢失风险必须使用通俗语言；
- 用户在任何阶段都能选择退出且不提交。

### 13.4 性能

- 首屏可交互时间目标小于 3 秒；
- 本地问题映射反馈目标小于 300 毫秒；
- 加密封存目标小于 2 秒，超时显示进度；
- 案例和咨询师推荐接口目标 P95 小于 1.5 秒。

## 14. 成功指标

### 14.1 漏斗指标

- 官网入口点击率；
- 方法说明完成率；
- 隐私声明后进入诊断率；
- 企业背景完成率；
- 自由问题有效输入率；
- 事件确认率；
- 问卷完成率；

- 案例卡阅读率；
- 咨询师卡阅读率；
- 主动展开咨询授权率；
- 最终授权提交率；
- 授权线索转初谈率；
- 初谈转项目率。

### 14.2 路由质量指标

- 企业接受候选事件比例；
- 企业保留候选题项比例；
- 顾问接受业务路由比例；
- 案例推荐被点击或展开比例；
- 推荐咨询师被人工保留比例；
- 最终项目与推荐服务一致率；

### 14.3 信任与隐私指标

- 隐私声明理解率；

## 15. 埋点需求

不得在分析埋点中记录企业原始问题、问卷答案、密钥、联系人或诊断明文。

允许的无内容埋点示例：

- `privacy_notice_viewed`；
- `vault_created`；
- `problem_mapping_generated`，仅记录候选数量和版本；
- `event_confirmed`，使用内部事件 ID；
- `question_mapping_removed`，使用题项 ID；
- `analysis_viewed`，记录覆盖模块数量；
- `case_card_viewed`，使用已授权案例 ID；
- `consultant_card_viewed`；
- `consulting_consent_opened`；
- `disclosure_preview_generated`；
- `disclosure_submitted`；
- `disclosure_revoked`。

## 16. 风险与应对

| 风险 | 后果 | 应对 |
|---|---|---|
| 用户设置弱口令 | 密文被离线破解 | 系统生成恢复密钥、口令强度检查、内存困难 KDF |
| 用户丢失密钥 | 数据不可恢复 | 明确提示、恢复码下载、企业多管理员恢复方案 |
| 网页脚本被篡改 | 明文或密钥被窃取 | CSP、依赖固定、供应链审计、独立安全评估 |
| 映射把现象当根因 | 错误咨询路由 | 输出机制假设、显示依据、企业与顾问双确认 |
| 案例相似度误导 | 企业形成效果预期 | 展示维度、证据口径和非承诺声明 |
| 未授权案例被展示 | 信任和合规风险 | 发布前授权校验、到期自动下架、审计告警 |
| 咨询申请被等同案例授权 | 非自愿公开 | 两次独立授权、不得默认勾选 |
| 模型供应商保留数据 | 数据控制失效 | 模型独立授权、本地/私有部署优先、合同与技术核验 |
| 顾问推荐忽略利益冲突 | 商业与职业风险 | 接入冲突登记和人工分配确认 |
| “零知识”宣传超出事实 | 误导客户 | 未审计前只描述具体技术事实，不使用绝对安全表述 |

## 17. P0 验收标准

1. 用户填写前可以看见数据处理、训练和访问声明；
2. 默认完成诊断不会向和君创建可读线索；
3. 服务器数据库中不存在诊断明文和企业主恢复密钥；
4. 使用错误密钥不能解密且不会破坏原密文；
5. 企业可在另一设备使用保险箱 ID 和恢复密钥恢复诊断；
6. 企业可选择提交字段和授权有效期；
7. 和君只能查看授权数据包内的字段；
8. 撤回后阻止后续在线访问并记录撤回事件；
9. 咨询申请不自动产生案例授权；
10. 客户侧只返回授权有效的案例；
11. 案例卡明确显示授权状态、匿名状态和指标口径；
12. 每个业务、案例和咨询师推荐均展示理由；
13. 无法匹配时返回信息不足和补充问题，不生成虚假推荐；
14. 关键操作具有时间、版本、主体和范围审计记录；
15. 前端 lint、production build、核心单元测试和端到端测试通过；
16. 上线前完成安全、隐私和法务评审。
